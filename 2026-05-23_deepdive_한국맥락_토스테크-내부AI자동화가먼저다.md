# 토스테크(toss.tech) — 고객 서비스보다 내부 자동화가 먼저다

## 누구/무엇인가

토스테크(toss.tech)는 한국 최대 핀테크 유니콘 비바리퍼블리카(토스)의 공식 엔지니어링 블로그입니다. 월간 활성 사용자 2,400만 명 이상의 슈퍼앱 토스를 운영하는 회사답게, 블로그에는 단순한 기술 소개가 아니라 "실제 프로덕션에서 무엇이 작동했는가"를 솔직하게 씁니다. 2025년 이후 토스테크에서 눈에 띄는 흐름은 하나입니다. **고객용 AI 기능보다 내부 개발자 생산성·운영 자동화용 AI를 먼저 구축했다**는 것입니다. Flowise 기반 에러 분석 자동화, RAG 챗봇 '박씨', LLM 취약점 분석 파이프라인이 모두 2025년 상반기에 집중적으로 공개되었습니다. 그 다음에야 고객 대면 AI 서비스인 'AI 어닝콜(AI Earnings Call)'이 등장했고, 출시 2주 만에 이용자 20만 명, 6개월 만에 120만 명을 넘겼습니다. 내부 역량 먼저, 외부 서비스 나중이라는 순서가 스피드의 비결이었습니다.

## 무엇이 특별한가

### 1. Flowise로 에러 분석을 LLM에 넘겼다 — "불친절한 에러 메시지" 문제 해결

토스 코어 Teens Growth Silo의 서버 개발자 조민규는 2025년 5월 이런 고통을 고백했습니다. "클라이언트 호출 에러가 발생하면, 보안 이슈로 추상화된 오류 메시지 탓에 본질적인 원인 파악에 불필요한 시간이 낭비됩니다." 해법은 요청 정보, API 스펙, 에러 스택 트레이스를 묶어 LLM에 던지는 파이프라인이었습니다. 도구는 Flowise — 코드 없이 드래그 앤 드롭으로 LLM 워크플로를 만드는 오픈소스 빌더입니다. 여기에 토스 자체 개발 모델인 **toss-standard**를 연결했습니다. 코드를 쓰지 않아도 LLM 에이전트를 조립할 수 있다는 점이 핵심입니다. 솔로 개발자에게 시사하는 바가 큽니다. Flowise는 n8n, Zapier AI 자동화와 유사하지만 온프레미스로 올릴 수 있어 데이터 유출 위험이 낮습니다.

### 2. RAG 챗봇 '박씨' — 문서를 쓰는 대신 문서가 찾아오게 했다

토스 프론트엔드 챕터가 직면한 문제는 많은 조직의 공통 고통입니다. "기술 문서가 생산자 입장에서 구조화되어 있어, 필요한 정보를 찾으려면 여러 번 탐색해야 한다." 그리고 "특정 팀원에게 지식이 몰려 같은 질문이 반복된다." 이를 해결한 것이 RAG(Retrieval-Augmented Generation) 기반 사내 지식 챗봇 **'박씨'**입니다. 이제 개발자는 문서를 뒤지는 대신 박씨에게 묻습니다. 박씨가 사내 문서 전체를 벡터 검색해 정확한 답을 가져옵니다. 토스 프론트엔드 팀은 "이제 지식이 자연스럽게 흐르는 환경이 됐다"고 표현했습니다. 지식 창고를 만드는 것이 아니라, 지식이 '능동적으로 찾아오는' 구조를 만든 것입니다.

### 3. LLM으로 보안 취약점을 자동 분석 — 100% 발견율, 비용은 오픈 모델로 해결

토스 보안팀은 LLM을 이용한 서비스 취약점 분석 자동화를 구축하며 두 가지 성과를 공개했습니다. 분석 경로 최적화 **50% 달성**, 취약점 발견 정확도 **100%** (단, 일부 false positive 포함). 그러나 수백 개 서비스를 분석하면 클라우드 LLM 비용이 월 수백만 원에 달할 것이라는 추산이 나왔습니다. 토스의 해결책은 **오픈 모델을 직접 호스팅**하는 것이었습니다. 이 결정 하나로 비용 대부분을 없앴습니다. GPU 클러스터 도입기(시리즈 #1, #2)를 별도 게시글로 낸 것도 이 맥락입니다. 자체 인프라 투자가 장기적으로 API 비용보다 낫다는 판단입니다.

### 4. AI 어닝콜: 내부 역량 → 고객 서비스로 전환, 2개월 제작 → 2주 20만 명

외부 AI 서비스의 결정판은 'AI 어닝콜'입니다. 해외 기업 실적 발표 오디오를 실시간으로 한국어로 번역·요약해주는 서비스로, 한국 증권사 최초였습니다. **단 두 달 만에 제작**했고, 출시 2주 만에 20만 명이 사용했습니다. 2025년 11월 기준 누적 이용자 120만 명을 돌파했고, 2,000건 이상의 어닝콜을 실시간으로 처리했습니다. 이어 'AI시그널' 서비스도 출시했습니다. 주가 변동이 통계적으로 의미 있는 수준인지 먼저 판단한 뒤, 그 원인을 AI가 설명하는 방식입니다. 속도가 가능했던 이유는 분명합니다. 내부에서 LLM 파이프라인을 쌓고, 자체 모델을 돌리고, Flowise로 에이전트를 조립해본 경험이 이미 쌓여 있었기 때문입니다.

### 5. toss-standard — 자체 모델을 만든다는 것의 의미

토스는 Flowise 파이프라인에 **toss-standard**라는 자체 LLM을 연결했습니다. 공개된 정보는 많지 않지만, 자체 GPU 클러스터를 도입하고 오픈 모델을 파인튜닝한 것으로 추정됩니다. 이는 단순한 OpenAI API 호출에 머무르지 않는다는 선언입니다. 금융 도메인 특화 데이터와 보안 요건 때문에 외부 클라우드 LLM 의존도를 낮추려는 전략입니다. 소규모 조직이라도 "어디까지 외부 API를 쓰고 어디서부터 자체 모델이 필요한가"를 생각하게 만드는 사례입니다.

## 와당탕/느린호밀 적용 포인트

**오늘 30분 (micro)**: 비서앱에서 반복되는 에러·문의 패턴 3가지를 메모에 적고, 그 중 하나를 LLM 프롬프트에 붙여 "원인 분석 + 해결안" 초안을 뽑아본다. 토스 Flowise 파이프라인의 핵심은 에러 컨텍스트(요청 정보 + API 스펙 + 스택 트레이스)를 한 번에 LLM에 넘기는 것입니다. 대표님도 ERP/비서앱 에러 로그를 Claude에 붙여넣고 "원인이 뭔지, 어떻게 고치면 되는지" 묻는 것부터 시작하면 됩니다.

**이번 주 1-2시간 (mid)**: 옵시디언에 "박씨 미니 버전" 세팅. 옵시디언 Smart Connections 플러그인(로컬 임베딩) + Copilot 플러그인 조합으로 와당탕·느린호밀 관련 노트를 검색 가능한 RAG 구조로 만들어봅니다. 아래 스니펫은 Claude API로 직접 사내 지식봇을 호출하는 예시입니다.

```python
import anthropic

client = anthropic.Anthropic()

def ask_knowledge_bot(question: str, context_docs: list[str]) -> str:
    context = "\n\n---\n\n".join(context_docs)
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="당신은 와당탕연구소·느린호밀의 내부 지식봇입니다. 제공된 내부 문서를 기반으로만 답하세요.",
        messages=[{
            "role": "user",
            "content": f"[내부 문서]\n{context}\n\n[질문]\n{question}"
        }]
    )
    return response.content[0].text

# 사용 예시
docs = ["ERP 주문 처리 플로우...", "비서앱 고객 응대 가이드..."]
print(ask_knowledge_bot("주문 취소 시 환불 처리 절차는?", docs))
```

**이번 달 실험 (macro)**: Flowise를 로컬 Docker로 설치하고, ERP에서 발생하는 에러 슬랙 알림을 Flowise 워크플로와 연결합니다. 에러 발생 → 컨텍스트 수집 → LLM 분석 → 슬랙에 "원인 + 해결 제안" 자동 게시 파이프라인을 구축합니다. 측정 지표: 에러 해결 평균 소요 시간(TTR, Time to Resolution) 30% 단축 여부.

## 한국 솔로 운영자 맥락에서 주의

**토스는 수천 명 조직이다.** GPU 클러스터, toss-standard 자체 모델, Flowise 내부 운영팀이 있기에 가능한 구조입니다. 솔로 운영자가 GPU 클러스터까지 따라갈 필요는 없습니다. 핵심만 뽑으면 됩니다. "에러 컨텍스트를 구조화해서 LLM에 던진다", "반복 질문을 RAG로 자동화한다" — 이 두 개면 충분합니다.

**데이터 보안 기준이 다르다.** 토스가 자체 LLM을 호스팅하는 이유 중 하나는 금융 규제 때문입니다. 와당탕연구소·느린호밀의 데이터가 외부 API로 나가는 것에 문제가 없다면, 오픈 모델 직접 호스팅보다 Claude API 활용이 훨씬 빠릅니다.
