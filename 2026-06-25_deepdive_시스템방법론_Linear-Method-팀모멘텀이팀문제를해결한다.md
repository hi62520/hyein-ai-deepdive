# Linear Method — "팀 모멘텀이 팀 문제를 해결한다"

## 누구/무엇인가

Linear Method는 이슈 트래킹 도구 Linear의 창업자 Karri Saarinen이 공개한 소프트웨어 개발 방법론입니다. Airbnb와 Coinbase에서 디자이너로 일했던 Saarinen이 2019년 창업 후 Linear를 키우면서 직접 체계화한 원칙들의 집합이고, linear.app/method에 전문 공개되어 있습니다. "기업용 지라 대체재"로 출발했지만 2026년 현재 엔터프라이즈 워크스페이스의 75%에 AI 에이전트가 설치된 AI 네이티브 협업 플랫폼으로 진화했습니다. 이 방법론이 주목받는 이유는 단순히 도구 사용법이 아니라, 속도와 품질을 대립 관계로 보지 않는 철학 때문입니다. 2026년 3월 Saarinen은 "이슈 트래킹은 죽었다"고 선언하며, Linear를 "컨텍스트를 실행으로 바꾸는 공유 제품 시스템"으로 재정의했습니다.

## 무엇이 특별한가

**1. "팀 모멘텀이 팀 문제를 해결한다"는 역발상**

Linear Method의 핵심 명제는 단순합니다. "Team momentum fixes team problems." 회의, 프로세스 정비, 조직 개편보다 개별 기여자가 매일 무언가를 완료하는 흐름 자체가 팀의 문제를 자연스럽게 해소한다는 관점입니다. 이를 위해 Linear는 2주 사이클을 표준으로 제시하는데, "짧아서 다른 우선순위를 놓치지 않고, 길어서 의미 있는 기능을 만들 수 있는" 황금 구간이라고 설명합니다. 이상적으로는 팀원 한 명이 한 주에 여러 항목을 완료할 수 있도록 작업을 잘게 쪼개야 하고, "생각만 하거나 이야기만 하는 것이 아니라, 매일 결정하고 앞으로 나아가야 한다"고 못 박습니다.

**2. 유저 스토리는 카고 컬트다**

Linear Method의 가장 도발적인 주장 중 하나는 유저 스토리 형식을 적극 배격한다는 점입니다. "User stories evolved twenty years ago, but modern customers are tech-savvy enough to articulate basic product requirements." 유저 스토리는 "작성하는 데도, 읽는 데도 시간이 과하게 들고 엔지니어를 기계적 역할로 가두는 우회적 업무 기술 방식"이라고 단언합니다. 대신 누구나 이해할 수 있는 평이한 언어로 "할 일"을 딱 한 줄로 기술하는 이슈 작성을 권장합니다. 이슈의 목적은 담당자가 실행하고, 팀원이 무슨 작업인지 이해할 수 있을 만큼 명확히 커뮤니케이션하는 것이지, 형식 준수가 아닙니다.

**3. 품질과 속도는 대립하지 않는다**

Saarinen은 "We don't believe that there is some kind of problem with doing both speed and quality. I think it's often they can actually go hand in hand if you have the right people."라고 말합니다. 이를 실현하는 구조적 장치가 "소규모 고품질 팀"입니다. 인원이 많아질수록 의견과 토론이 늘어 실행 품질이 희석된다는 전제 아래, 1~3명으로 1~3주 안에 완료되는 프로젝트 규모를 표준으로 설정했습니다. 또한 "디자인을 개발에 핸드오프"하는 방식을 명시적으로 거부하며, 모든 팀원이 품질에 공동 책임을 진다는 원칙을 세웠습니다.

**4. Continuous Planning: AI가 방법론을 바꿨다**

2025년 Linear는 "Continuous Planning"을 도입했습니다. 아이디어가 들어오는 즉시 "후보 프로젝트"로 정리하고, AI가 이를 유지·업데이트하며 패턴을 발견해 프로젝트를 자동 형성합니다. 플래닝 사이클이 시작될 때 "빈 화면 앞에 앉는" 시간 낭비가 사라집니다. 더 나아가 2026년 3월 기준 에이전트가 전체 이슈의 25%를 직접 생성하고 있으며, 엔터프라이즈 워크스페이스의 AI 에이전트 업무량은 3개월 만에 5배 성장했습니다.

**5. Opinionated Software의 철학**

Linear는 의도적으로 "의견이 있는 소프트웨어"를 만듭니다. 누구나 자신만의 워크플로를 발명할 수 있는 범용 도구가 아니라, "팀이 더 좋은 제품을 만들도록 돕는다"는 단일 목적에 최적화된 도구입니다. Saarinen은 "모든 사람이 각자의 워크플로를 발명하게 되면 결국 팀이 커질수록 혼돈이 온다"고 경고합니다. 이 철학이 Linear의 UX 결정 대부분을 설명합니다 — 불필요한 커스터마이징 옵션을 줄이고, 의도적으로 좁은 경험을 설계합니다.

## 와당탕/느린호밀 적용 포인트

**오늘 30분 (micro)**: 비서앱 ERP에서 이번 주 이슈 하나를 "유저 스토리" 형식 없이 작업 지시 한 줄로 재작성하고 실행까지 시간 측정. frontmatter의 `micro_action`과 동일.

**이번 주 1-2시간 (mid)**: 와당탕연구소 프로젝트 중 하나에 "Continuous Planning" 구조 적용. Claude Code 에이전트에게 요청이 들어오면 자동으로 ERP의 이슈 후보 리스트에 추가하는 훅 작성. 코드 스니펫 예시:

```python
# 비서앱 수신 요청 → ERP 이슈 후보 자동 생성
def create_candidate_issue(request_text, source="client"):
    issue = {
        "title": request_text[:80],   # 딱 한 줄, 유저 스토리 형식 X
        "status": "candidate",
        "source": source,
        "created_at": datetime.now().isoformat()
    }
    db.issues.insert(issue)
    return issue
```

**이번 달 실험 (macro)**: 와당탕 개발 사이클을 2주 단위로 재설정. 1~3명(대표님 + Claude Code 에이전트) × 1~3주 범위로만 프로젝트를 정의하는 규칙 수립. 매 사이클 끝에 "완료 이슈 수" 측정 — 모멘텀 지표 대시보드를 비서앱에 추가.

## 한국 솔로 운영자 맥락에서 주의

Linear Method는 "고품질 팀"을 전제합니다. 팀원 전원이 디자인과 개발 양쪽을 이해하고 공동 책임을 진다는 원칙은, 외주 개발자나 파트타임 협업자가 혼재하는 현실에서 그대로 적용하기 어렵습니다. 대표님처럼 Claude Code가 실질적 팀원 역할을 하는 경우라면, "에이전트에게 명확한 한 줄 이슈를 주고 완료 여부를 즉각 측정"하는 방식으로 번역해야 합니다 — 에이전트는 유저 스토리보다 명확한 지시어에 훨씬 잘 반응합니다.

또한 Linear Method의 2주 사이클은 팀 전체가 동기화된다는 전제 아래 작동합니다. 솔로 운영에서는 사이클 자체보다 "하루에 완료 가능한 단위로 이슈를 쪼개는 습관"이 더 실용적입니다. 매 사이클 리뷰 미팅 대신 옵시디언 데일리 노트에 "오늘 완료한 이슈 수"를 기록하는 것만으로 모멘텀을 체감할 수 있습니다.

- **훅**: "지라에서 매주 한 시간을 잃고 있다면, 당신이 쓰는 건 도구가 아니라 방법론 문제입니다 — Linear는 그걸 먼저 고쳤습니다."
