# Linear (Karri Saarinen) — "이슈 트래킹은 죽었다"와 컨텍스트 인프라 전략

## 누구/무엇인가

Linear는 2019년 핀란드 출신 디자이너 Karri Saarinen이 공동 창업한 소프트웨어 개발팀용 이슈 트래커입니다. Karri는 창업 전 Airbnb의 수석 디자이너로 디자인 시스템을 설계했고, Coinbase의 창업 디자이너이기도 했습니다. Jira가 장악하던 시장에 "빠르고, 키보드 중심이며, 의견이 분명한(opinionated)" 도구로 진입해 2025년 유니콘(기업가치 12.5억 달러)에 올랐습니다. 총 1억 3,420만 달러를 조달했으며 2025년 6월 Series C로만 8,200만 달러를 유치했습니다. 2026년 기준 178명의 인원으로 연간 반복 매출(ARR) 약 1억 달러를 달성하고 있습니다. 25,000개 이상의 팀이 사용 중이며, 2026년 3월 CEO가 "이슈 트래킹은 죽었다"고 선언하며 에이전트 플랫폼으로의 전환을 공식화했습니다.

## 무엇이 특별한가

**1. "이슈 트래킹은 죽었다" — 자기 시장을 스스로 부정한 선언**

2026년 3월 25일, Karri는 선언했습니다: *"Issue tracking is dead."* 이것은 경쟁사를 겨냥한 공격이 아니라 Linear 자신의 핵심 사업 모델에 대한 선제적 부정이었습니다. 그의 논리는 이렇습니다: *"에이전트가 버그를 5분에 고칠 수 있는데, 버그 보고 프로세스가 일주일이 걸린다면 — 실제 작업보다 프로세스가 훨씬 오래 걸리는 상황이 꽤 우스워 보이기 시작한다."* 에이전트가 실행을 장악하면, 인간의 병목은 "무엇을 만들지 결정하는 것"으로 이동합니다. Linear는 그 결정을 지원하는 컨텍스트 인프라로 포지셔닝을 바꿨습니다.

**2. 에이전트 채택 속도가 증거다 — 수치가 이미 말한다**

이 선언은 추측이 아니라 데이터에 기반합니다. Linear 엔터프라이즈 워크스페이스의 75% 이상에 에이전트가 설치되어 있습니다. 2026년 초 3개월 동안 에이전트가 처리한 작업량은 5배 증가했습니다. 그리고 가장 상징적인 수치: 신규 이슈의 25%는 이제 인간이 아니라 에이전트가 생성합니다. 인간이 만들던 작업 항목의 4분의 1을 이미 AI가 대신 쓰고 있다는 뜻입니다. OpenAI는 자사 내부 에이전트 오케스트레이션 시스템인 "Symphony"를 Linear 위에 구축할 만큼 이 플랫폼을 신뢰하고 있습니다.

**3. Linear Method — A/B 테스트 없이 취향으로 만든 도구**

Linear의 가장 비범한 점은 제품 개발 방법론입니다. Karri는 *"우리는 A/B 테스트를 하지 않는다. 메트릭 중심 의사결정을 하지 않는다. 취향과 판단으로 만든다"*고 말합니다. 대신 '의견이 분명한 소프트웨어(opinionated software)'를 추구합니다 — 좋은 기본값을 제공해 사용자가 설정에 낭비하는 시간을 없애는 방식입니다. 싸이클(Cycles)로 작업하고, 키보드 퍼스트로 설계하며, "취향 좋은 3-5명이 대규모 위원회보다 나은 결정을 내린다"는 원칙을 실천합니다. 178명으로 1억 달러 ARR을 만들고 있다는 사실이 이 철학의 생산성을 증명합니다.

**4. Claude Code + MCP — 터미널에서 Linear를 쓰는 워크플로**

Linear는 Claude Code를 위한 공식 MCP 서버를 제공합니다. 이 연동을 설정하면 Claude Code 에이전트가 Linear 이슈를 직접 조회·생성·업데이트할 수 있습니다. 실제 에이전트 루프 패턴은 이렇습니다: Linear에서 태스크 조회 → "In Progress"로 상태 변경 → 코드 구현 → 이슈에 코멘트 업데이트 → 인간 승인 체크포인트. PROGRESS.md를 에이전트 메모리로 사용하는 패턴도 결합되어, 멀티 세션에 걸친 작업 연속성을 확보할 수 있습니다. 터미널을 벗어나지 않으면서 이슈 관리를 자동화할 수 있는 현재 가장 성숙한 에이전트 워크플로 중 하나입니다.

**5. 자기복제를 노리는 "셀프드라이빙 Linear"**

Karri의 최종 비전은 Linear가 스스로 운영되는 시스템이 되는 것입니다. 버그가 들어오면 시스템이 자동으로 탐지하고 디버깅을 시작하며, 작업이 완료될 때만 인간에게 에스컬레이션합니다. 인간은 "결정"과 "검토"에만 집중하고 절차적 작업은 에이전트가 처리하는 구조입니다. 이를 위해 Linear는 Cursor와 Devin 같은 기존 코딩 에이전트를 감싸는 자체 코딩 에이전트를 개발 중입니다 — 더 깊은 플랫폼 통합으로 차별화하려는 전략입니다.

## 와당탕/느린호밀 적용 포인트

**오늘 30분 (micro)**: Claude Code의 `.claude/settings.json`(또는 MCP 설정 파일)에 Linear MCP 서버를 추가하고, "현재 In Progress 이슈 목록 보여줘"라는 자연어 명령 1개를 터미널에서 실행해보세요. `micro_action`과 동일: **Claude Code MCP 설정 파일에 Linear MCP 서버 항목을 추가해 비서앱 이슈를 터미널에서 자연어로 생성·조회하는 명령 1개 테스트**.

**이번 주 1-2시간 (mid)**: 비서앱 또는 ERP의 작업 항목을 Linear(또는 유사한 구조의 Notion/Obsidian 데이터베이스)에 연결하고, Claude Code 에이전트가 PROGRESS.md를 읽어 "오늘 처리할 이슈" 1개를 자동 선택 → 구현 → 완료 코멘트를 달도록 루프를 설계하세요. 코드 스니펫:
```bash
# Linear MCP 연동 후 Claude Code에서 실행
claude "Linear에서 현재 내 In Progress 이슈를 조회하고, 
첫 번째 이슈의 구현을 시작한 뒤 완료 시 코멘트를 달아줘"
```

**이번 달 실험 (macro)**: 와당탕연구소 ERP의 반복 작업(고객 문의 분류, 주간 보고서 생성, 콘텐츠 캘린더 업데이트) 중 하나를 에이전트가 Linear 이슈로 자동 생성하고 처리하는 파이프라인을 구축하세요. 측정 지표: 주당 에이전트가 생성한 이슈 수 vs 인간이 생성한 이슈 수. Linear의 목표인 "25% 에이전트 생성"을 본인 워크플로에서 재현하는 실험입니다.

## 한국 솔로 운영자 맥락에서 주의

**규모의 함정**: Linear는 25,000개 팀을 위한 도구입니다. 솔로 운영자에게 Jira 대신 Linear를 쓰는 것은 의미 있지만, 1인이 완전한 "Linear Agent" 생태계를 내부에 구축하는 것은 과투자일 수 있습니다. Linear의 에이전트 철학에서 배우되, 구현은 Claude Code + Obsidian + 간단한 마크다운 이슈 파일로 시작하세요. 도구가 아니라 패턴을 베끼는 것이 핵심입니다.

**"이슈 트래킹 죽음" 선언의 맥락**: 이 선언은 대규모 엔지니어링 팀의 절차적 오버헤드를 겨냥한 것입니다. 1인 운영자는 처음부터 이슈 트래킹이 가볍습니다. 솔로 맥락에서 진짜 시사점은 "에이전트가 실행을 맡으면 내가 집중해야 할 것은 무엇인가?"라는 질문입니다 — 와당탕/느린호밀의 경우 그것은 커리큘럼 설계, 고객 관계, 그리고 콘텐츠의 취향입니다.
